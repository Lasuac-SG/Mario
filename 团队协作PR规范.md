


## 一、 PR 标题规范 (与 Commit 规范一致)

PR 的标题必须能够准确概括本次修改的内容。我们采用标准的 `Type: 简明描述` 格式

**格式模板：** `type: 简短的变更描述` (注意冒号后面有一个空格) 
_示例：_ `feat: 新增用户微信登录接口` 
_示例：_ `fix: 修复个人主页头像无法加载的问题`

### 标准类型 (Type) 对照表

|   |   |   |
|---|---|---|
|**类型前缀**|**适用场景**|**示例**|
|**feat**|**(Feature)** 开发新功能|`feat: 购物车增加批量删除功能`|
|**fix**|**(Bugfix)** 修复程序 Bug|`fix: 修复订单金额计算错误的问题`|
|**docs**|**(Documentation)** 仅修改文档 (README 等)|`docs: 更新项目本地运行指南`|
|**style**|**(Style)** 不影响代码逻辑的格式修改 (空格、缩进)|`style: 格式化首页 CSS 代码`|
|**refactor**|**(Refactor)** 代码重构 (既不是新增功能也不是修复bug)|`refactor: 优化鉴权中间件的逻辑`|
|**perf**|**(Performance)** 提升性能的代码修改|`perf: 优化大数据列表的渲染速度`|
|**test**|**(Test)** 增加或修改测试用例|`test: 补充用户注册模块单元测试`|
|**chore**|**(Chore)** 构建过程、辅助工具、配置项的更改|`chore: 更新 .gitignore 排除日志文件`|

## 二、 分支命名规范

开发分支的命名应该与 PR 的类型保持对应，让其他人一眼就能看出这个分支的作用。

**格式模板：** `type/分支简短描述` (全部使用小写，单词间用连字符 `-` 分隔)

**常见示例：**

- 新功能分支：`feat/user-login`
    
- 修复分支：`bugfix/header-UI-bug` (注：分支名常用 bugfix 替代 fix)
    
- 重构分支：`refactor/api-network`
    
- 配置修改：`chore/update-webpack-config`
    

## 三、 PR 描述规范 (正文模板)

发起 PR 时，请不要留空正文描述（Description）。一段清晰的描述能帮审查者节省大量时间。建议直接复制以下模板进行填写：

```
### 1. 修改背景与目的
（简单说明为什么要做这次修改？如果是为了解决特定的 Bug，或者完成了某个需求，请在这里说明。）

### 2. 主要修改内容
- 新增了 xxx 模块
- 调整了 xxx 的逻辑
- 删除了废弃的 xxx 代码

### 3. 关联的 Issue（如果有）
（例如：Closes #12，合并后会自动关闭编号为 12 的 issue）

### 4. 测试与自查清单
- [ ] 代码已经在本地编译通过，功能运行正常。
- [ ] 已经自行 Review 过代码，没有遗留的调试代码（如 console.log / print）。
- [ ] 重要的逻辑已经添加了必要的代码注释。
```

## 四、 Commit 提交规范

Commit 是项目历史流转的基石。良好的 Commit 记录可以帮助我们在回滚代码和排查 Bug 时节省大量时间。

### 1. Commit 信息格式

我们采用业界通用的 Angular 规范（Conventional Commits）： `<type>(<scope>): <subject>`

- **type (必须):** 提交类型（对照下表）。
    
- **scope (可选):** 影响的范围，比如 `(user-api)`、`(login-page)`。
    
- **subject (必须):** 简短描述，说明做了什么事情。冒号后面**必须有一个空格**。
    

**示例：**

- `feat(auth): 新增用户微信登录接口`
    
- `fix(cart): 修复购物车金额计算错误的问题`
    

### 2. 标准类型 (Type) 对照表

|**类型前缀**|**适用场景**|
|---|---|
|**feat**|**(Feature)** 开发新功能|
|**fix**|**(Bugfix)** 修复程序 Bug|
|**docs**|**(Documentation)** 仅修改文档 (README 等)|
|**style**|**(Style)** 不影响代码逻辑的格式修改 (空格、缩进)|
|**refactor**|**(Refactor)** 代码重构 (既不是新增功能也不是修复bug)|
|**perf**|**(Performance)** 提升性能的代码修改|
|**test**|**(Test)** 增加或修改测试用例|
|**chore**|**(Chore)** 构建过程、辅助工具、依赖项的更改|

### 3. 提交颗粒度

- **小步提交：** 建议完成一个独立的小逻辑（如写完一个接口、切好一个页面组件）就 `git commit` 一次。**严禁**下班时把一天写的几千行代码用一个 `update` 打包提交。


## 五、 协作最佳实践（黄金法则）

1. **保持单一职责：** 一个 PR 只解决一件事（一个功能或一个 Bug）。**绝对不要**在一个 PR 里既开发 A 功能，又顺手重构 B 模块，还修复了 C 模块的 Bug。这会导致 PR 巨大，根本无法 Review。
    
2. **控制 PR 的大小：** 如果一个新功能特别大，请拆分成多个小的 PR 分步提交（例如：先提一个 PR 建表，再提一个写后端 API，最后提一个写前端页面）。
    
3. **先自我审查 (Self-Review)：** 在邀请别人看你的代码之前，请自己在 GitHub 的 `Files changed` 界面把代码从头到尾看一遍，确认没有低级错误。
